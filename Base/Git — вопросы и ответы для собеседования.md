## Middle

### Работа с историей

#### Что такое fast-forward merge?

Слияние, при котором Git просто **передвигает указатель ветки** вперёд, без создания merge-коммита. Это возможно, когда в целевой ветке (например, `main`) не было новых коммитов с момента, как от неё отделилась ветка-источник — то есть история линейна и конфликтов быть не может.

```bash
git checkout main
git merge feature   # если main не менялась — будет fast-forward
```

Чтобы принудительно создать merge-коммит даже там, где возможен fast-forward, используют `git merge --no-ff`.

#### Что такое squash merge?

Слияние, при котором **все коммиты фичи объединяются в один** новый коммит поверх целевой ветки. История фичи (промежуточные коммиты) в целевой ветке не сохраняется — остаётся один аккуратный коммит.

```bash
git merge --squash feature
git commit -m "Add feature X"
```

Плюсы: чистая линейная история в `main`. Минусы: теряется гранулярность истории конкретной фичи (хотя она остаётся в ветке фичи, если её не удалить).

#### Что делает git reset?

Перемещает указатель текущей ветки (HEAD) на другой коммит и, в зависимости от режима, меняет состояние индекса (staging area) и рабочей директории. Используется для отмены коммитов, снятия файлов со стейджа и т.д.

#### Чем git reset отличается от git revert?

- **`git reset`** — переписывает историю: перемещает HEAD/ветку на другой коммит, "удаляя" коммиты из истории ветки (сами объекты какое-то время остаются в базе, доступны через reflog). Опасен для уже запушенных веток, которые использует кто-то ещё.
- **`git revert`** — создаёт **новый** коммит, который отменяет изменения указанного коммита, не трогая существующую историю. Безопасен для публичных/общих веток.

#### В чём разница между --soft, --mixed и --hard?

| Режим | HEAD/ветка | Индекс (staging) | Рабочая директория |
|---|---|---|---|
| `--soft` | перемещается | не меняется (изменения остаются застейдженными) | не меняется |
| `--mixed` (по умолчанию) | перемещается | сбрасывается до состояния коммита (изменения расстейдживаются) | не меняется |
| `--hard` | перемещается | сбрасывается | сбрасывается — **все несохранённые изменения теряются** |

```bash
git reset --soft HEAD~1   # отменить коммит, оставить изменения застейдженными
git reset --mixed HEAD~1  # отменить коммит и стейдж
git reset --hard HEAD~1   # отменить коммит и все изменения безвозвратно
```

#### Что делает git restore?

Современная команда (Git 2.23+) для отмены изменений в рабочей директории и/или индексе, выделенная из перегруженного `checkout`.

```bash
git restore file.txt              # отменить изменения в рабочей директории
git restore --staged file.txt     # убрать файл из стейджа (аналог git reset file.txt)
git restore --source=HEAD~1 file.txt  # восстановить файл из другого коммита
```

#### Что делает git checkout?

Исторически универсальная команда с двумя разными задачами: переключение веток/коммитов **и** восстановление файлов. Это смешение обязанностей и стало причиной появления `switch` и `restore`.

```bash
git checkout main           # переключить ветку
git checkout HEAD~1 -- file.txt  # восстановить файл из коммита
```

#### Когда использовать git switch вместо checkout?

`git switch` (Git 2.23+) отвечает только за переключение и создание веток — это более безопасный и понятный аналог `checkout` без побочных эффектов работы с файлами.

```bash
git switch main
git switch -c feature/new-branch   # создать и переключиться
```

Рекомендуется использовать `switch`/`restore` вместо `checkout` в новом коде и обучении — меньше риска случайно потерять изменения.

#### Что такое git reflog?

Локальный журнал перемещений HEAD и веток (коммиты, resetы, rebase, checkout и т.д.), хранящийся отдельно от обычной истории. Позволяет **восстановить "потерянные" коммиты**, например после `reset --hard` или неудачного rebase, пока не отработал garbage collector.

```bash
git reflog
git reset --hard HEAD@{2}   # вернуться к состоянию до ошибки
```

Важно: reflog локальный и не пушится — не поможет, если данные потеряны на чужой машине.

#### Что такое git bisect?

Инструмент бинарного поиска коммита, который внёс баг. Вы указываете "хороший" и "плохой" коммит, а Git на каждом шаге предлагает промежуточный коммит для проверки, сокращая пространство поиска логарифмически.

```bash
git bisect start
git bisect bad                # текущий коммит плохой
git bisect good v1.2.0        # этот коммит был хороший
# Git чекаутит промежуточный коммит, вы тестируете и отвечаете:
git bisect good  # или
git bisect bad
# ... повторяется, пока Git не найдёт виновный коммит
git bisect reset
```

Можно автоматизировать с помощью `git bisect run <script>`, где скрипт возвращает код 0 (good) или 1 (bad).

#### Как найти коммит, который сломал приложение?

Основной инструмент — `git bisect` (см. выше). Дополнительно помогают:
- `git log -S"строка"` — найти коммиты, где менялось количество вхождений строки;
- `git log -p -- path/to/file` — просмотр истории изменений конкретного файла;
- `git blame` — кто и в каком коммите менял конкретную строку.

#### Как удалить последний коммит?

```bash
git reset --soft HEAD~1   # удалить коммит, оставить изменения застейдженными
git reset --hard HEAD~1   # удалить коммит и изменения полностью
```

Если коммит уже запушен — потребуется `git push --force-with-lease` (см. раздел про force push).

#### Как отменить несколько последних коммитов?

```bash
git reset --hard HEAD~3   # откатить 3 последних коммита (локально)
```
или безопасно для публичной истории:
```bash
git revert HEAD~3..HEAD   # создать revert-коммиты для последних трёх коммитов
```

#### Как изменить сообщение последнего коммита?

```bash
git commit --amend -m "Новое сообщение"
```
Если коммит уже запушен, потребуется `push --force-with-lease`, так как `amend` меняет SHA коммита.

#### Как изменить старый коммит?

Через интерактивный rebase:

```bash
git rebase -i HEAD~5
```
В открывшемся списке коммитов помечаем нужный как `edit`, Git останавливается на нём:
```bash
git commit --amend
git rebase --continue
```
Если менялось содержимое файлов, а не только сообщение, можно внести правки, `git add`, затем `--amend` и продолжить rebase. Важно помнить: это переписывает SHA всех последующих коммитов.

### Работа с изменениями

#### Что такое git stash и какие есть его возможности?

`git stash` временно "откладывает" незакоммиченные изменения (staged и unstaged) в отдельный стек, возвращая рабочую директорию к чистому состоянию последнего коммита. Полезно, когда нужно быстро переключиться на другую задачу.

Возможности:
```bash
git stash                      # застешить изменения
git stash -u                   # включая untracked файлы
git stash save "message"       # с описанием
git stash list                 # посмотреть список
git stash show -p stash@{0}    # посмотреть diff конкретного stash
git stash branch new-branch    # создать ветку из stash
git stash drop stash@{0}       # удалить конкретный stash
git stash clear                # удалить все stash
```

#### Что делает git stash pop?

Применяет последний (или указанный) stash к рабочей директории **и удаляет его** из стека, если применение прошло без конфликтов.

```bash
git stash pop
```

#### Чем stash apply отличается от stash pop?

- `git stash apply` — применяет изменения из stash, но **оставляет** его в стеке (можно применить повторно, например к нескольким веткам).
- `git stash pop` — применяет и **удаляет** stash из стека (эквивалент `apply` + `drop`).

Если при `pop`/`apply` возникает конфликт, stash не удаляется автоматически даже при `pop` — нужно разрешить конфликт и удалить stash вручную через `git stash drop`.

#### Что такое git clean?

Удаляет untracked-файлы из рабочей директории (файлы, которые не находятся под контролем версий и не в `.gitignore`, если не указано иное).

```bash
git clean -n     # dry-run: показать, что будет удалено
git clean -f      # удалить untracked-файлы
git clean -fd     # включая untracked-директории
git clean -fx     # включая файлы, игнорируемые через .gitignore
```

Команда деструктивна — рекомендуется всегда сначала запускать с `-n`.

### Работа с ветками

#### Как переименовать ветку?

```bash
git branch -m old-name new-name        # локально, если не находитесь в ветке
git branch -m new-name                 # если уже в переименовываемой ветке

# Обновить remote:
git push origin -u new-name
git push origin --delete old-name
```

#### Как удалить локальную и удалённую ветку?

```bash
git branch -d feature        # удалить локальную (безопасно, если смержена)
git branch -D feature        # принудительно удалить локальную
git push origin --delete feature   # удалить удалённую ветку
```

#### Что такое tracking branch?

Локальная ветка, привязанная к удалённой ветке (upstream), что позволяет использовать `git pull`/`git push` без указания remote и имени ветки, а также видеть, насколько локальная ветка опережает/отстаёт от удалённой.

```bash
git branch -u origin/main       # установить upstream для текущей ветки
git checkout --track origin/feature   # создать и привязать ветку одной командой
git branch -vv                  # посмотреть tracking-статус всех веток
```

#### Как обновить свою ветку относительно main?

Два основных подхода:

```bash
# 1. Merge — сохраняет всю историю, добавляет merge-коммит
git checkout feature
git merge main

# 2. Rebase — переносит коммиты фичи поверх актуального main, история линейна
git checkout feature
git rebase main
```

Rebase даёт более чистую линейную историю, но переписывает SHA коммитов фичи — не стоит делать rebase на ветке, которую уже используют другие (см. раздел Senior).

### Инструменты

#### Что такое git hooks?

Скрипты, которые Git автоматически запускает на определённых этапах workflow (перед коммитом, после коммита, перед push и т.д.). Хранятся в `.git/hooks/` и по умолчанию не версионируются вместе с репозиторием (поэтому для командного использования применяют инструменты вроде **Husky** или **pre-commit**, которые синхронизируют хуки через сам репозиторий).

Примеры хуков: `pre-commit`, `commit-msg`, `pre-push`, `post-merge`.

#### Что такое pre-commit hook?

Хук, запускаемый перед созданием коммита (до того, как открывается редактор сообщения коммита). Обычно используется для линтинга, форматирования кода, запуска быстрых тестов или проверки на секреты. Если скрипт завершается с ненулевым кодом — коммит прерывается.

```bash
#!/bin/sh
# .git/hooks/pre-commit
swiftlint --strict
```

#### Что такое git worktree?

Позволяет иметь **несколько рабочих директорий**, привязанных к одному репозиторию, каждая на своей ветке — без необходимости клонировать репозиторий заново или постоянно переключаться (stash/checkout) между ветками.

```bash
git worktree add ../hotfix-dir hotfix-branch
git worktree list
git worktree remove ../hotfix-dir
```

Удобно, например, для параллельной работы над hotfix, пока в основной директории идёт разработка фичи, или для запуска долгих тестов на отдельной ветке без прерывания текущей работы.

#### Что такое git submodule?

Механизм включения одного Git-репозитория внутрь другого как подкаталога, с фиксацией конкретного коммита внешнего репозитория. Полезно для переиспользования общих библиотек между проектами.

```bash
git submodule add https://github.com/org/lib.git Libs/lib
git submodule update --init --recursive   # при клонировании родительского репо
```

Минусы: усложняет workflow (нужно не забывать обновлять и коммитить ссылку на submodule), частая боль для новичков. Альтернативы — package managers (SPM, CocoaPods, Carthage для iOS) зачастую предпочтительнее для управления зависимостями.

#### Что такое Git LFS?

**Large File Storage** — расширение Git для эффективной работы с большими бинарными файлами (изображения высокого разрешения, видео, архивы, `.psd`, датасеты и т.п.). Вместо хранения самого файла в истории Git, LFS хранит лёгкий текстовый указатель (pointer), а сам файл — на отдельном сервере, загружаемый по требованию.

```bash
git lfs install
git lfs track "*.psd"
git add .gitattributes
git add design.psd
git commit -m "Add design file via LFS"
```

Актуально для iOS-проектов с большими ассетами (изображения, шрифты, ML-модели `.mlmodel`).

### Совместная разработка

#### Как вернуть коммит, который уже запушен?

Безопасный способ — `git revert`, создающий новый коммит, отменяющий изменения:

```bash
git revert <commit-hash>
git push
```

Для merge-коммита нужно указать родителя (`-m 1` обычно основная линия):
```bash
git revert -m 1 <merge-commit-hash>
```

`reset` + `force push` в этом случае избегают, если ветка публичная и её используют другие разработчики — это переписывает историю и ломает работу коллег.

#### Что делать, если возник merge conflict?

1. Git помечает конфликтующие места в файлах маркерами `<<<<<<<`, `=======`, `>>>>>>>`.
2. Открыть файлы, вручную выбрать/объединить нужный код, убрать маркеры.
3. Пометить файл как разрешённый: `git add file`.
4. Завершить merge: `git commit` (сообщение обычно генерируется автоматически).

```bash
git status              # посмотреть, какие файлы конфликтуют
# правим файлы
git add resolved-file.swift
git commit
```

Можно также использовать `git mergetool` для визуального разрешения конфликтов, или прервать процесс: `git merge --abort`.

#### Как разрешать конфликты после rebase?

Логика похожа на merge, но конфликты разрешаются **поэтапно, для каждого переносимого коммита**:

```bash
git rebase main
# конфликт — правим файлы
git add resolved-file.swift
git rebase --continue    # переходит к следующему коммиту
# при необходимости повторить
git rebase --abort        # прервать rebase и вернуться к исходному состоянию
git rebase --skip         # пропустить текущий коммит целиком (редко нужно)
```

Ключевое отличие от merge-конфликта: при rebase конфликты могут возникать несколько раз (по одному на каждый переносимый коммит), а не один раз.

#### Что такое Pull Request (Merge Request)?

Запрос на слияние изменений из одной ветки (обычно ветки фичи) в другую (обычно `main`/`develop`), создаваемый через веб-интерфейс хостинга (GitHub — Pull Request, GitLab/Bitbucket — Merge Request). PR — это не функция самого Git, а надстройка платформы, дающая пространство для код-ревью, обсуждения, автоматических проверок (CI), перед тем как изменения будут смержены.

#### Что такое Code Review?

Процесс проверки предложенных изменений другими разработчиками перед их слиянием в основную ветку. Цели: найти баги и проблемы дизайна на раннем этапе, поддержать единый стиль кода, обменяться знаниями о кодовой базе, снизить bus factor. Обычно реализуется через PR/MR с комментариями построчно и обязательными approve перед merge.

### Workflow

#### Что такое GitFlow?

Модель ветвления с фиксированными долгоживущими ветками:
- `main`/`master` — только релизные версии;
- `develop` — основная ветка разработки;
- `feature/*` — ветки фич, ответвляются от `develop`;
- `release/*` — подготовка релиза (баг-фиксы, версионирование) перед мержем в `main` и `develop`;
- `hotfix/*` — срочные фиксы прод-версии, ответвляются от `main`.

Плюсы: чёткая структура, удобна для проектов с плановыми релизами (например, мобильные приложения с релизными циклами через App Store). Минусы: избыточная сложность для команд с непрерывной поставкой (CI/CD, много деплоев в день).

#### Что такое trunk-based development?

Модель, при которой все разработчики работают в одной общей ветке (`trunk`/`main`) с очень короткоживущими feature-ветками (часы-дни), которые быстро мержатся обратно. Незавершённый функционал скрывают через feature flags, а не через долгоживущие ветки. Считается стандартом для команд с CI/CD и частыми релизами.

#### Что такое feature branch workflow?

Каждая новая фича или фикс разрабатывается в отдельной ветке, ответвлённой от основной (`main`/`develop`), а после завершения и код-ревью сливается обратно через Pull Request. Это базовый, наиболее распространённый workflow, лежащий в основе как GitFlow, так и более лёгких схем (например, GitHub Flow — упрощённая версия без `develop` и `release` веток).

## Senior

### Стратегии работы

#### Когда использовать merge vs rebase vs squash?

- **Merge** — когда важно сохранить полную историю и контекст того, как и когда ветки объединялись; безопасен для публичных веток, не переписывает историю.
- **Rebase** — когда нужна чистая линейная история без лишних merge-коммитов; хорош для приватных/локальных веток перед их публикацией.
- **Squash** — когда промежуточные коммиты фичи (фиксы опечаток, "wip", "fix review comments") не несут ценности и в `main` должен попасть один осмысленный коммит на фичу/задачу.

Частая практика: rebase на свою ветку локально для актуализации + squash merge при финальном слиянии PR в `main`.

#### Когда нельзя делать rebase?

Главное правило: **никогда не делать rebase публичной/общей ветки**, которую уже подтянули (pull) другие разработчики. Rebase переписывает SHA коммитов, и если кто-то уже основал свою работу на старых коммитах, после rebase их история разойдётся, что приведёт к дублированию коммитов и путанице при следующем merge/pull.

Безопасно рибейзить только: свои локальные/личные ветки, которые ещё не были запушены или которыми точно никто не пользуется кроме вас.

#### Когда допустим force push?

Допустим на:
- собственной feature-ветке, которой не пользуется никто другой (после rebase/amend);
- ветке PR до код-ревью или по согласованию с командой (например, после squash перед мержем).

Практически никогда не допустим на:
- `main`/`master`, `develop`, релизных ветках без крайне веской причины и согласования со всей командой.

#### Что такое --force-with-lease и почему он безопаснее --force?

`git push --force` перезаписывает удалённую ветку безусловно, даже если кто-то другой успел запушить туда изменения после того, как вы в последний раз получали (fetch) состояние ветки — вы рискуете затереть чужую работу.

`git push --force-with-lease` перед пушем проверяет, что состояние удалённой ветки на сервере совпадает с тем, что вы ожидаете (то есть с тем, что вы видели в последнем fetch). Если кто-то успел запушить туда что-то ещё — push отклоняется с ошибкой, и вы не затираете чужие изменения без предупреждения.

```bash
git push --force-with-lease origin feature
```

#### Как организовать git workflow для большой команды?

Ключевые элементы:
- Выбор модели ветвления под ритм релизов: GitFlow — для плановых релизов, trunk-based/GitHub Flow — для непрерывной поставки.
- Обязательные Pull Request с код-ревью и статус-чеками CI (тесты, линтеры, сборка) перед мержем.
- Защищённые ветки (branch protection): запрет прямого push в `main`, запрет force push, минимальное число approve.
- Единый стиль коммитов (например, Conventional Commits) для читаемой истории и автогенерации changelog.
- Squash или rebase-merge стратегия для чистой линейной истории в `main`.
- Автоматизация через git hooks / CI (линтинг, форматирование, проверка на секреты в коммитах).
- Чёткий процесс для hotfix и release-веток, отдельный от обычного feature-флоу.

#### Как организовать release branch?

Ветка `release/x.y.z` создаётся от `develop` (в GitFlow) или от `main` (в упрощённых схемах) в момент "заморозки" функционала перед релизом. В неё вносятся только багфиксы, обновление версии, финальная стабилизация — новые фичи не добавляются. После готовности релиз мержится и в `main` (с тегом версии), и обратно в `develop`, чтобы фиксы попали в основную линию разработки.

```bash
git checkout -b release/1.4.0 develop
# багфиксы, bump version
git checkout main
git merge --no-ff release/1.4.0
git tag -a v1.4.0 -m "Release 1.4.0"
git checkout develop
git merge --no-ff release/1.4.0
git branch -d release/1.4.0
```

#### Как работать с hotfix?

Ветка `hotfix/*` создаётся напрямую от `main` (текущего прод-релиза), минуя `develop`, чтобы срочно исправить критичный баг без ожидания следующего планового релиза. После фикса она мержится обратно и в `main` (с новым патч-тегом), и в `develop`/актуальные release-ветки, чтобы исправление не потерялось при следующем релизе.

```bash
git checkout -b hotfix/1.4.1 main
# исправление бага
git checkout main
git merge --no-ff hotfix/1.4.1
git tag -a v1.4.1 -m "Hotfix 1.4.1"
git checkout develop
git merge --no-ff hotfix/1.4.1
git branch -d hotfix/1.4.1
```

### Внутреннее устройство Git

#### Как Git хранит данные?

Git — это, по сути, **content-addressable файловая система** поверх графа объектов. Каждый объект (blob, tree, commit, tag) хранится в базе объектов (`.git/objects`), адресуется по хэшу своего содержимого, и объекты образуют направленный ациклический граф (DAG): коммиты ссылаются на дерево файлов и на родительские коммиты, деревья — на blob'ы и другие деревья.

#### Что такое blob?

Объект, хранящий **содержимое файла** (без имени и метаданных — только сырые данные). Если два файла с разным именем имеют идентичное содержимое, они будут ссылаться на один и тот же blob.

#### Что такое tree?

Объект, представляющий **директорию**: список записей вида "имя файла/поддиректории + режим доступа + ссылка на blob или другой tree". Именно tree связывает содержимое (blob) со структурой файловой системы и именами файлов.

#### Что такое commit object?

Объект, фиксирующий снимок проекта в определённый момент времени. Содержит: ссылку на корневой `tree` (состояние всех файлов), ссылку на один или несколько родительских коммитов (для merge-коммитов — несколько), автора, коммитера, дату и сообщение коммита.

#### Что такое tag object?

Отдельный тип объекта, создаваемый для **аннотированных тегов** (`git tag -a`). В отличие от лёгкого тега (просто именованная ссылка на коммит), tag object хранит собственные метаданные: автора тега, дату, сообщение, и опционально — GPG-подпись. Ссылается на коммит (обычно), но может ссылаться и на другой объект.

#### Что такое SHA-1 (или SHA-256 в новых версиях Git)?

Криптографическая хэш-функция, используемая для вычисления уникального идентификатора каждого объекта Git на основе его содержимого. Именно этот хэш служит "адресом" объекта в content-addressable хранилище. Исторически Git использовал SHA-1; из-за обнаруженных теоретических уязвимостей коллизий SHA-1 (например, атака SHAttered) современный Git поддерживает переход на SHA-256 как более криптостойкую альтернативу (хотя SHA-1 остаётся дефолтом в большинстве репозиториев по историческим причинам и вопросам совместимости).

#### Почему Git называют content-addressable filesystem?

Потому что каждый объект в базе данных Git адресуется не по имени файла или пути, а по хэшу собственного содержимого. Изменилось содержимое — изменился хэш — это фактически новый объект. Такая модель гарантирует целостность данных (любое повреждение сразу обнаруживается по несовпадению хэша) и естественную дедупликацию (одинаковое содержимое хранится один раз, независимо от того, в скольких местах/коммитах оно встречается).

#### Что такое pack files?

Способ компактного хранения множества объектов Git в одном файле с дельта-компрессией (объекты хранятся не целиком, а как разница относительно похожего объекта) плюс zlib-сжатие. Используется для оптимизации размера репозитория и скорости передачи по сети (при `push`/`fetch`/`clone` данные передаются в виде packfile). Создаются автоматически при `git gc` или упаковке во время push.

#### Что такое loose objects?

Отдельные несжатые (точнее, сжатые только zlib, без дельта-компрессии) файлы-объекты, которые Git создаёт по одному на каждый новый blob/tree/commit сразу после операций вроде `commit`. Со временем их накапливается много, и `git gc` объединяет их в более компактные pack files.

#### Что делает garbage collection (git gc)?

Оптимизирует репозиторий: упаковывает loose objects в pack files, удаляет объекты, которые больше не достижимы ни из одной ссылки (веток, тегов) и старше периода хранения (по умолчанию около 2 недель, настраивается через `gc.pruneExpire`), очищает устаревшие записи reflog. Запускается автоматически время от времени, но можно вызвать и вручную:

```bash
git gc              # обычная оптимизация
git gc --aggressive # более тщательная (и медленная) оптимизация
git gc --prune=now  # немедленно удалить недостижимые объекты
```

#### Что делает git fsck?

**File System Check** — проверяет целостность и связность объектной базы Git: ищет повреждённые объекты, "битые" ссылки, недостижимые (dangling) коммиты/blob'ы, которые не привязаны ни к одной ветке или тегу.

```bash
git fsck --full
git fsck --lost-found   # найти "потерянные" коммиты/blob'ы
```

Полезен для диагностики повреждения репозитория и для поиска данных, потерянных после агрессивного `reset --hard` или удаления веток, если они уже вышли из reflog.

### Производительность

#### Почему большой репозиторий начинает тормозить?

Основные причины:
- Огромное количество коммитов/объектов — растёт время выполнения операций, требующих обхода истории (`log`, `blame`, `bisect`).
- Большие бинарные файлы в истории (изображения, видео, архивы), которые не были вынесены в Git LFS — они многократно дублируются в истории при каждом изменении и раздувают объём `.git`.
- Много loose objects без регулярного `git gc` — не хватает дельта-компрессии.
- Очень много веток/тегов — замедляет операции со ссылками и некоторые сетевые команды.
- Большое количество файлов в рабочей директории — замедляет `status`, `checkout`, индексацию (особенно на файловых системах без hardware/OS-акселерации, актуально для крупных монорепозиториев).

#### Как уменьшить размер репозитория?

- Перевести крупные бинарные файлы на **Git LFS**.
- Удалить большие файлы из истории (`git filter-repo` — см. ниже), если они туда случайно попали.
- Регулярно выполнять `git gc --aggressive` для оптимальной упаковки объектов.
- Использовать `shallow clone`/`partial clone` там, где не нужна полная история (например, в CI).
- Разбить монорепозиторий на несколько репозиториев/использовать submodules, если это оправдано архитектурно.

#### Как удалить большой файл из истории Git?

Современный рекомендуемый инструмент — **`git filter-repo`** (заменил устаревший и медленный `git filter-branch`):

```bash
git filter-repo --path path/to/big-file.zip --invert-paths
```

Альтернатива для простых случаев — **BFG Repo-Cleaner**:
```bash
bfg --delete-files big-file.zip
```

После очистки истории необходимо:
```bash
git push --force --all
git push --force --tags
```
и предупредить всю команду о необходимости заново склонировать репозиторий, так как SHA всех коммитов после изменённого файла изменятся.

#### Когда использовать Git LFS?

Когда в проекте регулярно появляются/изменяются крупные бинарные файлы, которые плохо поддаются дельта-компрессии Git (изображения, видео, аудио, архивы, ML-модели, дизайн-файлы `.sketch`/`.psd`). Для iOS-проектов это часто ассеты высокого разрешения, шрифты, `.mlmodel` файлы. LFS не подходит для файлов, которые редко меняются и не слишком велики — там overhead LFS (отдельный сервер, pointer-файлы) не оправдан.

#### Что делает shallow clone (--depth)?

Клонирует репозиторий, ограничивая глубину истории — скачивается только указанное количество последних коммитов, без полной истории проекта.

```bash
git clone --depth 1 https://github.com/org/repo.git
```

Плюсы: значительно быстрее и компактнее — полезно для CI/CD, где не нужна вся история, только текущее состояние кода. Минусы: недоступны операции, требующие полной истории (`git log` вглубь, `git blame` на старые коммиты, `bisect` за пределами загруженной глубины). Можно "дозагрузить" историю позже: `git fetch --unshallow`.

### Практические задачи

#### Как перенести несколько коммитов в другую ветку?

Через `git cherry-pick` с диапазоном коммитов:

```bash
git checkout target-branch
git cherry-pick <commit1> <commit2> <commit3>
# или диапазон (не включая первый коммит):
git cherry-pick commitA^..commitB
```

Альтернативный подход для большого числа последовательных коммитов — `git rebase --onto`:
```bash
git rebase --onto target-branch source-branch~3 source-branch
```

#### Как разбить один большой коммит на несколько?

```bash
git rebase -i <commit>~1     # пометить коммит как "edit"
git reset HEAD~1              # расстейджить изменения коммита, оставив их в рабочей директории
# теперь добавляем и коммитим изменения частями:
git add file1.swift
git commit -m "Первая логическая часть"
git add file2.swift
git commit -m "Вторая логическая часть"
git rebase --continue
```

#### Как объединить несколько коммитов?

Через интерактивный rebase со `squash`/`fixup`:

```bash
git rebase -i HEAD~5
```
В открывшемся редакторе для всех коммитов, кроме первого, меняем `pick` на `squash` (сохранить сообщение) или `fixup` (отбросить сообщение), затем сохраняем — Git объединит их в один коммит.

#### Как полностью переписать историю репозитория?

Инструмент — **`git filter-repo`** (современная замена `filter-branch`), позволяющий переписать всю историю: удалить файлы/директории, изменить пути, заменить email/имена авторов, обрезать историю и т.д.

```bash
git filter-repo --path secrets.env --invert-paths
git filter-repo --mailmap mailmap.txt
```

После такой операции все SHA коммитов меняются, требуется `push --force` и уведомление всей команды о необходимости пересоздать локальные клоны.

#### Как восстановить удалённую ветку?

Если ветка была удалена локально, но её SHA известен (например, из `reflog` или из истории терминала):

```bash
git reflog                       # найти SHA последнего коммита удалённой ветки
git branch recovered-branch <sha>
```

Если ветка была удалена на удалённом репозитории (например, после merge PR), но кто-то из команды имеет её локально или в своём reflog/fetch-истории:
```bash
git fetch origin
git checkout -b recovered-branch origin/deleted-branch   # если ещё не удалено полностью на сервере
```
На GitHub/GitLab также часто можно восстановить недавно удалённую ветку через веб-интерфейс (ссылка "restore branch" сразу после мержа PR).

#### Как восстановить случайно удалённый коммит?

```bash
git reflog                 # найти SHA коммита до удаления
git cherry-pick <sha>       # перенести коммит в текущую ветку
# или
git branch recovered <sha>  # создать ветку на этом коммите
```

Если коммит уже "потерян" из reflog (истёк срок хранения) — можно попробовать `git fsck --lost-found`, который найдёт недостижимые, но ещё не удалённые объекты.

#### Как найти, где впервые появился определённый код?

```bash
git log -S"строка_или_код" --oneline           # найти коммиты, где менялось число вхождений строки (pickaxe)
git log -G"регулярное_выражение" --oneline     # то же, но с regex-поиском по диффам
git log --follow -- path/to/file               # история файла с учётом переименований
```

Для более узкого случая ("когда впервые эта строка появилась в файле") часто комбинируют с `git blame` и последующим `git log -p` по конкретному коммиту.

#### Как найти автора конкретной строки (git blame)?

```bash
git blame path/to/file.swift
git blame -L 10,20 path/to/file.swift    # только строки 10–20
git blame -w path/to/file.swift          # игнорировать изменения только в пробелах
git blame -C path/to/file.swift          # учитывать перемещение кода между файлами
```

Показывает, каким коммитом, автором и когда была добавлена/изменена каждая строка файла.

#### Как откатить merge commit?

Обычный `git revert` не работает напрямую с merge-коммитами без указания, какого родителя считать "основным" — так как у merge-коммита несколько родителей и Git не знает, какую линию изменений отменять.

```bash
git revert -m 1 <merge-commit-sha>
```
`-m 1` означает "оставить изменения от первого родителя как базовую линию" (обычно это та ветка, в которую вливались, например `main`), отменяя изменения, привнесённые вторым родителем (влитой веткой).

Важный нюанс: после такого revert повторный merge той же ветки в будущем (даже с новыми коммитами) может не подхватить старые изменения автоматически — Git считает, что они уже были "отменены". Для повторного включения тех изменений иногда требуется сначала сделать revert самого revert-коммита.

#### Как перенести репозиторий на другой remote без потери истории?

```bash
git remote -v                      # посмотреть текущие remote
git remote set-url origin <новый-url>   # изменить URL, если репозиторий на новом хостинге тот же

# или, для полного клона со всеми ветками и тегами на новый remote:
git clone --mirror old-repo-url
cd old-repo.git
git push --mirror new-repo-url
```

`--mirror` гарантирует перенос всех веток, тегов и refs, включая полную историю, без необходимости пересоздавать коммиты.

### Практические вопросы (очень любят на собеседованиях)

#### Ты случайно сделал git reset --hard. Что можно сделать?

Проверить `git reflog` — там будет запись о состоянии HEAD до reset. Найти нужный SHA и восстановиться:

```bash
git reflog
git reset --hard HEAD@{1}   # или конкретный SHA из reflog
```

Если изменения не были закоммичены вообще (просто рабочая директория), reflog не поможет — они, скорее всего, потеряны безвозвратно, если только не остались, например, в кэше редактора/IDE.

#### Запушил секретный API-ключ. Как исправить ситуацию?

Порядок действий:
1. **Немедленно отозвать/сменить сам ключ** на стороне сервиса — это самое важное действие, так как ключ уже мог "утечь" (даже приватный репозиторий не гарантия — история могла быть закэширована, форкнута, просмотрена ботами).
2. Удалить ключ из истории Git с помощью `git filter-repo` (или BFG Repo-Cleaner).
3. `git push --force` в удалённый репозиторий и уведомить команду о необходимости пересоздать локальные копии.
4. Добавить файл с секретами в `.gitignore`, а сами секреты вынести в переменные окружения / secret manager, чтобы предотвратить повтор.
5. Рассмотреть pre-commit хуки для проверки на секреты (например, `git-secrets`, `gitleaks`) на будущее.

#### После rebase появились конфликты. Что делать дальше?

Разрешить конфликт для текущего коммита в процессе rebase, затем продолжить:

```bash
# правим конфликтующие файлы
git add resolved-file.swift
git rebase --continue
```
Если конфликтов слишком много или что-то пошло не так — можно безопасно откатиться к состоянию до rebase:
```bash
git rebase --abort
```

#### Сделал 15 мелких коммитов. Как превратить их в один?

```bash
git rebase -i HEAD~15
```
Оставить `pick` у первого коммита, у остальных 14 заменить на `squash` (или `fixup`, если промежуточные сообщения не нужны), сохранить — Git объединит все 15 в один коммит с возможностью отредактировать итоговое сообщение.

Более быстрый вариант для этого конкретного случая:
```bash
git reset --soft HEAD~15
git commit -m "Единое сообщение коммита"
```

#### Нужно перенести один коммит из ветки А в ветку B.

```bash
git checkout branchB
git cherry-pick <commit-sha-из-branchA>
```
При конфликте — разрешить его как обычно, затем `git cherry-pick --continue` (или `--abort`, чтобы отменить).

#### Нужно удалить файл из всей истории Git.

```bash
git filter-repo --path path/to/file --invert-paths
git push --force --all
git push --force --tags
```
Обязательно уведомить команду о необходимости пересоздать локальные клоны, так как история переписывается полностью (меняются все SHA после первого появления файла).

#### После git pull появились лишние merge-коммиты. Почему?

`git pull` по умолчанию делает `fetch` + `merge`. Если в локальной ветке уже есть коммиты, которых нет на удалённой, а на удалённой появились новые коммиты — Git не может сделать fast-forward и автоматически создаёт merge-коммит для объединения двух разошедшихся линий истории.

Решения:
```bash
git pull --rebase        # вместо merge — переносит локальные коммиты поверх удалённых
git config --global pull.rebase true   # сделать rebase поведением по умолчанию для pull
```

#### Как откатить релиз, который уже находится в production?

Зависит от инфраструктуры, но с точки зрения Git обычно:
1. Если это просто откат кода (а не отдельный процесс деплоя/rollback CD-системы) — используют `git revert` для соответствующих коммитов/merge-коммита релиза, создавая новый коммит отмены, а не переписывание истории (`main` — публичная ветка).
2. Заново собирают и деплоят на основе состояния после revert.
3. Часто в реальности откат делается не через Git вовсе, а через redeploy предыдущего артефакта/образа (Docker image, .ipa/App Store откат к предыдущей версии) — Git-история используется скорее для последующего анализа и подготовки правильного фикса.

#### Что делать, если случайно сделал git push --force в main?

1. Немедленно сообщить команде, чтобы никто не продолжал работать поверх испорченной истории.
2. Найти правильный SHA до неудачного force push — через `git reflog` (если он есть у вас или у коллеги, кто недавно синхронизировался), либо через историю CI/CD системы, либо через reflog на сервере (у некоторых self-hosted решений есть доступ к server-side reflog).
3. Восстановить ветку:
```bash
git push --force-with-lease origin correct-sha:main
```
4. В будущем — включить **branch protection** на `main` (запрет force push и прямых push вообще), чтобы такая ситуация была невозможна на уровне настроек репозитория.

#### Как восстановить ветку, удалённую несколько дней назад?

Порядок действий:
1. Проверить `git reflog` — если кто-то (вы или коллега) недавно работал с этой веткой локально, её SHA может там сохраниться (с учётом срока хранения reflog, по умолчанию 90 дней для достижимых и 30 для недостижимых записей).
2. Проверить хостинг (GitHub/GitLab/Bitbucket) — многие платформы хранят возможность восстановления недавно удалённой ветки через UI, особенно сразу после merge PR.
3. Если ветка была смержена перед удалением (стандартная практика после PR) — код уже находится в целевой ветке, восстанавливать саму ветку как таковую может и не требоваться.
4. В крайнем случае — `git fsck --lost-found` для поиска недостижимых объектов, если reflog уже "протух".
